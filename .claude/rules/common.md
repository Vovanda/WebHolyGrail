# Common rules — R0–R14 для Holy Grail

> Развёрнутые правила. Краткая шпаргалка — в `/CLAUDE.md`. Исходник — в Memory MCP `HolyGrail/30 — Философия и архитектура`.

## R0. Хардкод на фронте = антипаттерн

Корневая ценность Holy Grail. **Данные сайта живут в БД (Payload), не в коде.** Граница: «это про _данные_ или про _реализацию_?». Данные → CMS. Реализация (JSX-скелет, токены/стили, логика рендера, типы) → код.

Если в админке **нет соответствующего поля** для правки опечатки / переставки секции / добавления-удаления элемента / скрытия блока — значит контент захардкожен в коде. Это PR-блокер: добавить Payload-поле и убрать хардкод до коммита в `main`.

Дефолты в коде допустимы **только как graceful fallback** на пустое поле / недоступную CMS, не как источник правды.

## R1. Не верстать страницы голыми утилитами

Tailwind-утилиты — **внутри компонентов**, наружу — типизированный React-интерфейс. Кнопка существует **в одном экземпляре**, один источник правды.

**Плохо:**

```tsx
<div className="px-6 py-4 bg-blue-500 rounded-lg text-white font-bold">Заказать</div>
```

**Хорошо:**

```tsx
<Button variant="primary" size="lg">
  Заказать
</Button>
```

Если контрол отсутствует — сначала его, потом использование. Не «один раз можно».

## R2. Токены конечны

Цвет, отступ, радиус, тень, шрифт — **только через CSS-переменные** из `styles/tokens.css`. Никаких `bg-[#3a7bd5]`, `mt-[13px]`, inline-color.

**Tailwind config** наследуется от токенов:

```ts
theme: {
  colors: { primary: 'var(--color-primary)', ... },
  spacing: { sm: 'var(--space-sm)', ... }
}
```

Исключение — рантайм-значения (`style={{ '--progress': value + '%' }}`), это R6.

## R3. Изоляция фронта и бэка через `contracts`

`client/` **не импортирует** ничего из `cms/`. Граница — папка `contracts/` над `src/`. Зависимость **однонаправленная**: `client/` / `cms/` / `api/` → `contracts/`. Обратной нет.

**Плохо:** `import { Page } from '../../cms/collections/Pages'` в client.
**Хорошо:** `import { Page } from 'contracts/pages'`.

CMS отдаёт данные **в форме `contracts/`**, не в форме своих внутренних типов.

## R4. Масштабирование сбоку

Новая фича = **новый сервис / новый модуль / новая коллекция + блок к нему**. Ядро не трогается.

Появилась бизнес-логика «расчёт стоимости вязки»? → `api/` рядом с `cms/`, не добавлять endpoint к Payload. CMS остаётся рулить контентом.

## R5. Блок = чистая функция от пропсов

На **любой глубине** дерева. Блок:

- получает пропсы, рендерит детей;
- **не лезет в родителя**;
- **не тянет данные сам** (это делает страница, передаёт пропсами);
- **не знает где стоит**.

Это позволяет рекурсивно вкладывать блоки и переставлять структуру не трогая содержимое.

**Контрпример (SP Page Builder):** блок третьего уровня обращается к настройкам родителя второго уровня. Год спустя — комок без переезда.

### R5+ Блок совместим с будущим визуальным конструктором (Уровень 3)

> Конструктор уровня 3 (см. Memory `HolyGrail/36`) — отдельный далёкий продукт. **Сейчас его не строим (R9, R7).** Но блоки, которые пишем уже на старте, должны быть **архитектурно пригодны** к тому чтобы их в нём поднимать. Иначе придётся переписывать сотни блоков перед тем как запустить конструктор. Это закладка через **дисциплину пропсов**, не через инфраструктуру.

**Что это значит на практике для каждого блока:**

1. **Пропсы — JSON-сериализуемые.** Только примитивы, массивы, объекты, ссылки на media (id/url), ссылки на другие блоки (id). **Никаких** `() => void`-callback'ов в основном API блока (события — отдельный слой, конструктор их подцепит сам), **никаких** instance'ов классов, Date-объектов в сыром виде, Map/Set, React-нод в пропсах (если нужны слоты — массив-описание блоков, не `ReactNode`).
2. **Один контракт пропсов = одна типизированная схема в `contracts/`.** Конструктор будущего читает этот тип и автоматически рисует редактор полей. Если пропсы непредставимы как schema — блок не выйдет в редактор.
3. **Никаких скрытых зависимостей.** Блок не лезет в `useRouter`, `useSearchParams`, глобальный store, `window`, `document` (вне `useEffect`-измерений). Всё что нужно — приходит в пропсах. Иначе при выдёргивании в canvas-редактор блок сломается без контекста страницы.
4. **Стили самодостаточны.** Блок не зависит от CSS родителя (нет правил типа «работает только внутри `.hero-section`»). Токены — глобальные, это нормально. Каскадные хаки — нет.
5. **Понятные имена пропсов на русском в meta.** Имя в коде — английское (`titleAr`, `ctaLabel`), но в `meta` для редактора — человеческое русское (`'Заголовок'`, `'Текст кнопки'`). Это для директора в будущем редакторе.
6. **Каждый блок экспортирует meta-описание** рядом с компонентом:
   ```ts
   export const HeroMeta = {
     name: 'Hero',
     displayName: 'Главный экран',
     category: 'content',
     schema: HeroPropsSchema,   // тот же тип что в contracts/
     defaultProps: { ... },
     preview: '/blocks/hero/preview.png',  // когда появится
   } as const;
   ```
   Это **уже сейчас**. Конструктор когда-нибудь подцепит, нам ничего переделывать не придётся. Без meta — блок «слепой» для будущего редактора.
7. **layout vs content разделение (см. `35`).** Layout-блоки (Grid/Stack/Columns) принимают `children: BlockNode[]` — массив **описаний** блоков, не React-нод. Content-блоки — листья, без `children`.

**Что НЕ делать (закладывает мину под будущий конструктор):**

- Принимать `children: React.ReactNode` в основном API блока.
- Использовать `cloneElement`, render-props, compound-components внутри блока (`<Tabs><Tabs.Tab/></Tabs>`) — это паттерны для разработчика, не для сериализации.
- Завязывать поведение на URL, query-params, cookies из самого блока.
- Хранить состояние блока во внешнем глобальном store через свободно выбранные ключи. Если состояние нужно — это пропс + опциональный callback.

Эта дисциплина **не делает блок сложнее писать** — наоборот, заставляет сразу думать о нём как о чистом компоненте. Цена нулевая, выигрыш — через год не переписываем 50 блоков перед запуском canvas-редактора.

### R5++ Именование блоков — функциональное, не доменное

**Мы пишем шаблон будущего, не сайт-для-veo55.** Каждое имя компонента в `client/blocks/` (и в `packages/ui/`) должно проходить тест:

> «Назвал бы я его так же, если бы делал сайт для **автосервиса** / **кофейни** / **клиники**? Если нет — имя плохое.»

**Правильно (функциональные имена):**

- `EntityCard`, `EntityGrid`, `EntityPair`
- `DetailDialog`, `Lightbox`, `Carousel`
- `Hero`, `CTAStrip`, `Timeline`, `Checklist`
- `BadgeCount`, `Tag`, `Pill`
- `ContactsBlock`, `FormBlock` (а не `BookingForm` — booking это уже домен)

**Неправильно (доменные/клиент-специфичные имена):**

- ❌ `DogCard`, `LitterGrid`, `PuppyCounter`, `DogModal`
- ❌ `MenuItem` (для кофейни), `AppointmentCard` (для клиники), `ServiceTile` (для автосервиса)

**Где доменные имена ОК:**

- **Payload-коллекции** (`Dogs`, `Litters`, `Owners`) — это **данные** конкретного клиента, не визуальный компонент. Имя описывает что хранится. Это нишевая часть (см. memo `HolyGrail/38`).
- **Страницы маршрутов** в `app/(site)/dogs/page.tsx` — URL отражает домен, это нормально.
- **`client/blocks/niche/`** — последнее место для блока который **действительно** никак не обобщается (и это редкий случай — обычно можно подобрать функциональное имя).

**Если возникает соблазн назвать `DogCard`:**

1. Спросить — что в нём специфично для собак? Если только данные (картинка, имя, бейджи) — это `EntityCard` с пропсами `image, title, badges, href`.
2. Если визуальный паттерн «А × Б» (родители) — это `EntityPair` с `left, right, separator`.
3. Если кажется что блок «совсем нишевый» — **поднажми обобщение ещё на уровень.** Родословная собаки? → `GenealogyTree` (применимо: семейное древо людей, орг-структура, иерархия товаров, цепочка предков любых сущностей). Карта помёта? → `RelationGraph`. Расписание занятий? → `ScheduleGrid`. Почти всё обобщается.

**Правило прохождения через PR-ревью:** в имени `*.tsx` блока встретилось слово `Dog`, `Litter`, `Puppy`, `Owner` — это блокер. Сначала пробуем обобщить (как выше — на 1-2 уровня абстракции вверх). Только если **доказано** что обобщение невозможно (а это редкость) — переезд в `niche/` с явной аргументацией.

**`client/blocks/niche/` должна быть почти пустой.** Если она пухнет — значит обобщения не дожимаем.

## R5+++ Темизация — через токены, не через `dark:` Tailwind

**Тема = override CSS-переменных через `data-theme="<name>"` атрибут на `<html>`.** Компонент пишет `bg-bg text-ink` всегда — меняется только что эти токены значат.

- ❌ **Никаких** `bg-white`, `bg-black`, `text-gray-900`, `bg-slate-50` — даже белый/чёрный приходят из токенов (`bg-bg`, `text-ink`).
- ❌ **Никаких** `dark:` Tailwind-prefix'ов в компонентах — это второй параллельный канал темизации, дублирующий tokens.
- ✅ Только `bg-bg`, `bg-surface`, `text-ink`, `text-muted`, `border-border`, `bg-accent`, etc — имена из `tokens.css`.
- ✅ Темы добавляются в `tokens.css` как `:root[data-theme='dark'] { ... }` блоки. Контракт `SiteSettings.theme.mode = 'light' | 'dark' | 'auto'` управляет какой блок применяется.

**Если в shadcn-копии встретилось `bg-background text-foreground` / `bg-primary text-primary-foreground`** — переименовать на наши имена при копировании. Никаких параллельных систем токенов в проекте.

**Гарантия:** при включении любой темы (dark, contrast, sepia) **ни один компонент не переписывается**.

## R13. URL — ЧПУ для сущностей, query-params где оправдано

**Дефолт для маршрутов сущностей — slug в пути, не id в query.**

- ✅ `/dog/zorka-veo` — карточка собаки по slug
- ✅ `/litter/n-2026` — помёт «литера Н 2026»
- ✅ `/news/sertifikat-rkf` — новость по slug
- ✅ `/about`, `/faq`, `/contacts` — статичные страницы
- ❌ `/dog?id=123` — id в query там где есть естественный slug
- ❌ `/page?slug=about` — параметризация пути там где можно быть в иерархии

**Query-params уместны и нормальны для:**

- **Состояния списка** — `/catalog?name=А&p=2`, `/dogs?color=cheprachny&sex=female&sort=name`, `/blog?tag=release&page=3`. Это разные **view одной коллекции**, не разные сущности.
- **Cross-cutting флаги** — `?utm_source=vk&utm_campaign=puppies-2026`, `?preview=true` (draft-mode), `?debug=1` (DX-флаг).
- **Share-links с предзаданным фильтром** — посетитель шлёт ссылку «вот эти 3 щенка», state в query.

**Принцип:**

- **Сущность с собственной "карточкой"** → её путь = ЧПУ (`/dog/<slug>`, `/litter/<slug>`).
- **Список с применённой выборкой** → ЧПУ для коллекции + query для фильтров (`/catalog?...`).
- **Не плодить два пути к одной странице** (`/dog/zorka-veo` И `/dog?slug=zorka-veo` — выбрать один, второй редиректить).

**Для SEO** ЧПУ выигрывает на странице-карточке. Для коллекционных view (фильтры) Google рекомендует **canonical** на «чистый» URL без фильтра + `noindex` на отфильтрованные view.

**В Next App Router** это значит:

- `app/(site)/dog/[slug]/page.tsx` — карточка через ЧПУ.
- `app/(site)/catalog/page.tsx` + `searchParams` для фильтров.
- Никаких `app/(site)/page-by-id/page.tsx` с `?id=`.

## R14. SSR по умолчанию, client-side fetching только когда оправдан

**Next 15 + Server Components — дефолтное поведение в Holy Grail.**

- ✅ Server Component с `async function Page()` + `await getX()` — данные приходят на сервере, HTML рендерится сразу с контентом, SEO видит всё.
- ❌ Client Component с `useEffect → fetch → useState(data)` — SEO видит пустой shell, FCP/LCP страдает, гидратация прыгает.

**`'use client'` нужен только когда:**

- Реальный пользовательский input (`<input onChange>`, форма с валидацией)
- Эффект на DOM (focus-trap, scroll-lock, IntersectionObserver, анимация на mount)
- Подписка на browser API (`localStorage`, `matchMedia`, `navigator`)
- Сторонняя библиотека требует client (большинство визуальных UI-либ)

**Что НЕ повод делать `'use client'`:**

- «Хочу хук» — большинство хуков можно заменить серверной логикой.
- «Хочу обработчик клика на Link» — Next `<Link>` работает без `'use client'`.
- «Хочу `useState` для toggle» — если состояние малое и не критичное, можно через HTML `<details>` / CSS-only `:has()`.

**Граница:** если блок может быть server-component — он **обязан** быть им. Переход в client = осознанное решение с причиной.

## R6. Без догм

- **Inline `style`** — для рантайм-динамики (`style={{ '--col': cols }}`).
- **Utility-классы прямо в JSX** — для скорости, **если это не публичный блок** (если блок, то R1).
- **Раздельный CSS** — для сложных переносимых контролов.

Инструмент — по бусту, не по идеологии. Но R1/R2/R5 — не отменяются.

## R7. Целевая нагрузка на старте — нулевая

У всех новых сайтов трафика нет. **Никакого highload авансом:**

- Без BFF / API-route в `client/app/api/` пока не нужен (фронт читает напрямую из CMS через `contracts/`).
- Без кэшей / Redis / реплик авансом.
- Без отдельного `api/` пока бизнес-логика не появилась.
- Без `_template` пока второй сайт не показал что повторять.

Highload-готовность — следствие чистой архитектуры, добавляется по факту.

## R8. Доступ к данным инкапсулирован

К БД **никто не лезет напрямую** в обход слоя доступа.

- `cms/` (Payload) → Local API / Drizzle. Свои репозитории-обёртки **не делать** (обёртка над обёрткой).
- `api/` (когда появится) → репозитории по Clean Architecture (интерфейс в домене, реализация в инфраструктуре).
- Конкретная БД (SQLite/Postgres) — деталь реализации, меняется адаптером в `payload.config.ts`.

Важно **наличие границы**, не единая терминология.

## R9. Абстракция следует за опытом

**Снизу вверх. Не наоборот.**

- Сначала **один работающий veo55**.
- Потом из него **вычитывается `_template`** (в отдельной ветке, чтобы main оставался живым).
- Когда два-три сайта на `_template` — то что в них повторяется уходит в **`packages/ui` и `packages/contracts`**.
- Когда сборка нового сайта стала ритуалом — **скиллы-генераторы** (`new-site`, `add-block`).

Строить «сразу как надо» = зафиксировать вслепую решения, которые потом переделывать.

**Порядок:** конкретное → обобщение → автоматизация.

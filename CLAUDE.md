# Web Holy Grail — корневой инструктаж агенту

> Это проект **фреймворка** для производства честных сайтов микробизнесу — не один сайт, а архитектура + `_template` под серию. Сейчас в проекте один живой сайт `sites/veo55` — **прототип/обкатка идеи**, временно-главная задача. Не путать: цель проекта — `_template`, veo55 — испытательный стенд.

## Цель и тест-критерий

**Цель:** архитектура + `_template`, из которого следующие сайты (автосервис, кофейня, клиника, …) собираются за часы, не недели.

**Тест-критерий:** _«если проект правдив — аналог veo55 на этой архитектуре собирается за несколько часов»_. Каждое решение фильтруется вопросом: **«это сделает следующий сайт быстрее или медленнее?»**

**Что я делаю сейчас на veo55:** «точно скопировать с улучшениями» — у заказчика повезло, есть готовый старый сайт + данные + референс. Для большинства следующих сайтов референса не будет — буду проектировать сам (см. `holygrail-ui-reference` skill режим B).

## Что значит «Holy Grail-уровень» — не «работает же», а чисто, осознанно, поддерживаемо

Проект называется Holy Grail не случайно. Уровень исполнения:

- **SOLID и Clean Architecture — ежедневная дисциплина.** SRP в каждом модуле/компоненте, DIP на границах слоёв, изоляция домена от инфраструктуры. Карточка, парсер, JSON-эндпоинт, блок — каждый делает **одно**. Не лепить всё в один файл, не дублировать, не копировать 200 строк CSS между компонентами.
- **Думаем принципами, проектируем до кода.** Сначала разобрать: откуда данные, какие side effects, что ломается, кто owner, на каком слое решается. Применять паттерны (адаптеры, фасады, репозитории, фабрики) там где они уместны — не «сел и начал кодить».
- **Лучше дольше, но корректно.** Скорость берётся из того что **не приходится переделывать**. Если архитектурно правильный путь дольше — иди им. Auto mode ≠ быстро без проверки.
- **Не быдлокодить — читать доку.** Не знаю как у Payload устроены globals / у Next работает middleware / у Drizzle мигрируется схема → читаю официальную (через `WebFetch`). Plausible-sounding объяснение без проверки = пиздёж с умным видом, Володя ловит мгновенно.
- **Один owner — всё через код / CLI / API.** Не «зайди в админку, сделай X», не «накатишь миграцию?». Ручной шаг чужими руками = техдолг которым некому заниматься.
- **Без хаков и редиректов «здесь там везде».** `next.config redirect`, `nginx rewrite`, `.htaccess RewriteRule`, костыли в CSS `!important` — невидимый слой грязи на год. Архитектурно правильный путь дольше, но **поддерживается**.
- **Anti-make-busy.** Не решать «левые» проблемы из видимости работы. Не «без рефакторинга X я не починю Y» без доказательства.
- **Sparring partner, не yes-man.** Вижу что Володя предложил то что сломает Z — встреваю с конкретным аргументом (файл:строка, какой контракт ломается). Молчаливое поддакивание = потеря доверия. Без пруфов спорить с senior'ом бессмысленно.
- **Не сторонний наблюдатель.** Вижу что плохо — встреваю сразу, не жду пока спросит.

### Качество кода — без скидок на «это всего лишь UI»

UI-код часто пишут плохо. **У нас так не пишут.** Блоки в обозримом будущем станут материалом для визуального конструктора (Уровень 3) — писать сегодня надо так, чтобы потом не переписывать сотни компонентов.

- **SRP в компонентах.** Один блок — одна ответственность. Не «универсальный Hero с 30 пропсами и тремя if'ами» — это будущий комок.
- **Чистые функции от пропсов (R5).** Никаких скрытых `useContext` без явного интерфейса, никаких прыжков в глобальный store из листа дерева.
- **Никаких `any`, никаких `// @ts-ignore`** без `// FIXME:` и тикета. Типы — часть контракта.
- **Магические числа / захардкоженные строки → токены (R2), константы, словари.**
- **Дубль третий раз → выносится в общее** (не на втором — R9, но на третьем точно).
- **Техдолг — честно и сразу.** Ввёл хак → `// TODO(holygrail-<num>):` + отдельная задача в PLAN. Не «потом вспомним». Если знаю что надо рефакторить чтобы фича села правильно — рефакторю сейчас.
- **Тесты — в том же коммите**, где появилась логика. Контракты, рендер блоков, валидация форм, нишевая логика — юнит-тест сразу. Не «следующим PR». Покрываем то что **реально может сломаться**.
- **Документация в коде — обязательна для публичного API.** TSDoc на типах `contracts/`, Payload-коллекциях, экспортируемых функциях, React-компонентах: что делает, что принимает, side effects. Нетривиальные алгоритмы — коммент **ПОЧЕМУ так**. Хаки/workaround'ы — обязательно с причиной и условием когда можно убрать. Внутренние частные функции — только если WHY не очевиден из имени.

## Рабочий цикл сессии (операционный ритм после `/load-context`)

1. **Контекст в памяти.** Источник правды — MCP `HolyGrail/session/active` (что в работе СЕЙЧАС) + `HolyGrail/plan/PLAN` (этапы и статусы). Не держать состояние только в голове.
2. **План пунктами линейно.** Этапы (A/B/C) → подэтапы (A1/A2) → атомарные действия. Каждый пункт читабелен без контекста.
3. **Расширение плана — только при новом запросе Володи.** Встроить в существующий план там где логично по приоритету. Не «переписать всё». «Заодно почищу X» = backlog, не активный фокус.
4. **Приоритизация явная.** Один **фокус №1** в верху файла. Всё остальное упорядочено, помечено, но НЕ конкурирует за внимание сейчас. Новая задача — либо встраивается в фокус, либо ниже, никогда «вровень».
5. **Не скакать.** Линейно: A1 → smoke → A2 → smoke → … Володя прерывает корректировкой посреди этапа — **актуализировать план + продолжить текущее**, не пивотировать на половине фразы. Если корректировка требует поменять активный пункт — менять явно, отметив в PLAN «доделано до X, переключаюсь на Y по запросу [time]».
6. **Cron auto-tick прогресса** — создаю в начале сессии (off-minutes, не `:00/:30`).
7. **Сверка цель vs факт.** На каждом завершённом этапе — короткий self-review: «что заявлено в PLAN ↔ что реально в коде/БД/UI». Несовпадения — либо доделать, либо обновить PLAN с честной отметкой «откатил / не сделал / отложил» + причина. **Не отмечать done если факт не совпадает.**

## Стек

- **Frontend:** Next 14+ (App Router, SSR), React, shadcn/ui copy в `components/ui/`, Tailwind, CSS-токены
- **CMS:** Payload 3.x (TypeScript, MIT, self-host), русская админка
- **БД:** SQLite на старте, Postgres когда понадобится. Адаптер в `payload.config.ts`
- **Контейнеризация:** Docker. Dockerfile у `client/` и `cms/`. Compose в `deploy/local/`
- **Тесты:** Playwright + Vitest

## Stop-conditions (зови Володю)

1. Архитектурная развилка (новая модель данных, breaking contract, выбор стека)
2. Зацикливаюсь — один fix не работает второй раз
3. Деструктивная операция на shared/prod (force-push, удаление веток, `rm db`)
4. Что-то выглядит подозрительно

## Что НЕ делать

- **CMS внутри Next.** Payload — отдельное приложение с собственным Dockerfile
- **Прямые импорты `cms/` → `client/`.** Только через `contracts/` (R3)
- **`bg-[#hex]` / inline-color.** Только токены (R2)
- **Доменные имена блоков** (`DogCard`, `LitterGrid`). Только функциональные (R5++)
- **Деструктивные op на проде без явного «ок»**
- **Изобретать обобщения раньше второго случая (R9)**
- **Выдуманные данные** в реальную БД. Только реальные или очевидно-фейковые префиксы (`test-`, `TEST_`, `lorem`)

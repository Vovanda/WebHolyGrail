# Web Holy Grail — корневой инструктаж агенту

> Кодовая база Web Holy Grail монорепо. Сюда заходит человек или Claude перед тем как трогать код. Архитектурный контекст — в Memory MCP, проект **HolyGrail** (внутри MCP сохранилось короткое имя — это историческое; смысловое имя проекта — Web Holy Grail), заметки 00–90. Здесь — рабочие правила и навигация.

## Что это

**Web Holy Grail** — opinionated dev-среда и растущая база компонентов для производства честных сайтов микробизнесу. Не конструктор, не визуальный билдер.

### Цель проекта (главное — не перепутать)

**Цель — Web Holy Grail: архитектура + `_template`, из которого следующие сайты собираются за часы, не недели.** Это первичный результат, ради которого взялись.

**veo55 — реалистичная нагрузка, проверяющая архитектуру на вшивость.** Тест-критерий простой и жёсткий: _«если проект правдив — аналог veo55 на этой архитектуре собирается за несколько часов»._ Если собирается за неделю или с матерком — архитектура где-то лжёт и надо пересматривать. Каждое решение по дороге фильтруется вопросом: **«это сделает следующий сайт быстрее или медленнее?»**

**Оля (мама-кинолог, владелица veo55.ru) получает сайт попутно** — приятный побочный эффект, но не мотивация. Запрос клиента, для понимания: «отвяжите от Joomla, визуально можно копию текущего сайта; если переведёте на современные компоненты — отлично».

### Из чего что вырастает

- **Сейчас:** один живой `sites/veo55` (в `main`), строится С САМОГО НАЧАЛА в дисциплине «думаю как будет извлекаться в шаблон». Не «как соберу — потом подумаю», а **сразу со взглядом на `_template`**.
- **После того как veo55 живой и зелёный:** в отдельной ветке (`template/extract-from-veo55`) вычитывается `_template` — всё инвариантное переезжает в `packages/_template`, в `sites/veo55/` остаётся только нишевое (Dogs, Litters) и уникальное (брендинг, тексты).
- **Дальше:** второй клиент собирается из `_template` за обещанные часы. Это и есть проверка-на-вшивость в боевых условиях.

### Архитектурно

Монорепо-зонтик (см. `32`), сайт = `client/` (Next, SSR) + `cms/` (Payload, **отдельное приложение**) + `contracts/` (разъём), при росте — `api/` (см. `33`).

## Что Володя ждёт от проекта (вне зависимости от R1–R9)

Проект называется **Web Holy Grail** не случайно. Уровень исполнения — не «работает же», а **чисто, осознанно, поддерживаемо**. Это означает:

- **SOLID и Clean Architecture — не лозунг, а ежедневная дисциплина.** SRP в каждом модуле, DIP на границах слоёв, изоляция домена от инфраструктуры. Карточка, парсер, JSON-эндпоинт, блок — каждый делает **одно**. Не лепить всё в один файл. Не дублировать. Не копировать 200 строк CSS между компонентами.
- **Системное мышление — до кода.** Сначала разобрать: откуда данные, какие side effects, что ломается, кто владелец, на каком слое решается. Потом писать. «Сел и начал кодить» → запах того что не подумал.
- **Лучше дольше, но корректно.** Auto mode ≠ быстро без проверки. Скорость берётся из того что **не приходится переделывать**, а не из того что быстро накидал. Если архитектурно правильный путь дольше — иди им.
- **Не быдлокодить — читать доку.** Если не знаю как у Payload устроены globals / у Next работает middleware / у Drizzle мигрируется схема — **читаю официальную документацию**, не сочиняю plausible-sounding объяснение. Plausible без проверки = пиздёж с умным видом, Володя ловит и наказывает.
- **Один owner — всё делаю сам.** Не «зайди в админку, сделай X», не «попроси Володю накатить миграцию» — делаю через код / CLI / API. Каждый ручной шаг, который не я — техдолг, которым потом некому заниматься.
- **Не плодить хаки и редиректы «здесь там везде».** `next.config redirect`, `nginx rewrite`, `.htaccess RewriteRule`, костыли в `CSS !important` — все эти штуки потом забываются и образуют невидимый слой грязи. Архитектурно правильный путь дольше, но **поддерживается**.
- **Anti-make-busy.** Не решать «левые» проблемы из видимости работы. Не «без рефакторинга X я не починю Y» без доказательства. Если задача — починить Y, чиню Y.
- **Sparring partner, не yes-man.** Если вижу что Володя предложил то что сломает Z — встреваю с конкретным аргументом («это сломает inv flow в Pages, переделывать день»), не молча соглашаюсь. Молчаливое поддакивание = потеря доверия. Без пруфов спорить нельзя — растёрт в порошок. Формула: позиция = основание + готовность отстоять.
- **Не сторонний наблюдатель.** Вижу что плохо — встреваю сразу, не жду пока спросит.

### Качество кода — без скидок на «это всего лишь UI»

UI-код часто пишут плохо («ну это же вёрстка, она и так одноразовая»). **У нас так не пишут.** Дополнительный аргумент: **блоки в обозримом будущем станут материалом для визуального конструктора** (Уровень 3, см. Memory `HolyGrail/36`). Сейчас его не строим (R9), но писать блоки уже сегодня надо так, чтобы потом не переписывать сотни компонентов перед запуском canvas-редактора. Чистота сейчас = возможность собрать конструктор позже без переделки.

Конкретные требования к блоку-под-будущий-конструктор — в `.claude/rules/common.md` секция «R5+». Главное: **пропсы JSON-сериализуемы, нет скрытых зависимостей от контекста страницы, есть `meta`-описание для редактора**.

Компонент-блок, страница, форма, hook — всё подпадает под те же требования что domain и infrastructure:

- **SRP в компонентах.** Один блок — одна ответственность. Не лепить «универсальный Hero» с 30 пропсами и тремя if'ами внутри — это будущий комок.
- **Чистые функции от пропсов (R5).** Никаких скрытых `useContext` без явного интерфейса, никаких прыжков в глобальный store из листа дерева, никакого «знаю что родитель — это X».
- **Никаких `any`, никаких `// @ts-ignore` без `// FIXME:` и тикета.** Типы — часть контракта, не декорация.
- **Магические числа / захардкоженные строки** → токены (R2), константы, словари.
- **Дубль третий раз → выносится в общее** (не на втором случае — R9, но на третьем уже точно).

### Данные — только правдивые, не выдуманные

**В реальный проект реального клиента (даже в dev-БД) нельзя класть выдуманные имена / регалии / характеристики собак, щенков, людей.** Это:

- **R0-нарушение** (контент должен быть из правды, не из головы)
- **Клевета на питомник**, если случайно увидит человек или попадёт в кеш/индексацию
- **Подмена правды** — выдуманные данные смотрятся как настоящие и потом не отличишь

**Правила для seed/fixtures:**

1. **Сначала ищу реальные данные на veo55.ru / в `C:\...\veo55\`**. Если можно достать — беру оттуда.
2. **Если реальных данных нет под рукой и они нужны для smoke** — либо спросить Володю явно, либо использовать **очевидно-фальшивые префиксы**: `test-do-not-use-mother`, `TEST_TITLE_2099`, `lorem ipsum sit amet`. Чтобы при любом взгляде было видно «это не данные, это структурный плейсхолдер».
3. **Никогда** не использовать правдоподобные звучащие имена/регалии/места («Чемпион РКФ», «Москва, Евразия 2024», «Барс») — это в любой момент может оказаться неправдой или чьей-то реальной собакой.
4. **Smoke-данные = либо реальные, либо очевидно-тестовые.** Третьего не дано.

### Skills checklist — что обязательно активировать

Прежде чем взять задачу — проверить applicable skills (раздел «available skills» в system reminder) и **активировать подходящий**, не идти на дефолте:

- **UI / новый блок / страница / визуальная композиция** → `frontend-design` (предотвращает generic-AI-look — без него уйдёшь в shadcn-дефолт)
- **Меняешь Payload-коллекцию / блок / global** → `payload-migration` (правильный workflow миграции SQLite, не сломать prod)
- **Кропать фото собак** → `veo-photo-crop`
- **Перенос-1:1 / face-off с оригиналом** → `simplify` + `feedback_visual_compare.md` из памяти
- **UI-механика (carousel, dnd, charts, forms)** → НЕ велосипедить, берём battle-tested (см. `feedback_no_velociped_use_opensource.md`): Embla, react-hook-form, etc.

Если задача попадает в зону скилла и скилл **не активирован** — это автоматический фак-ап. Активация скилла занимает 1 вызов, экономит часы переделок.

### Инструменты для насмотренности и документации — обязательны, не из памяти

Любой Hero/Carousel/Form в голове превращается в усреднённый shadcn-default 2024 года. **Правила (auto-fail при нарушении — это самые часто-нарушаемые правила, см. жалобу «иногда посредственный UI»):**

1. **Нет открытого референса → не пишу JSX.** Поймал себя на «помню что hero/timeline/feed делается так» — стоп. `mcp__playwright__browser_navigate` на оригинал или `WebSearch` для насмотренности.
2. **Нет открытой доки API → не вызываю метод чужой библиотеки.** Поймал на «помню что у Embla опции это `{loop, align}`» — `WebFetch('https://www.embla-carousel.com/api/options/')` и сверка.
3. **Перед коммитом UI — Playwright screenshot моего рендера в основных breakpoint'ах (1920 / 768 / 375) + сравнение с оригиналом скрином-в-скрин.** Не «выглядит ок на глаз», не «макет в голове совпал».
4. **`frontend-design` skill активирован** до первой строки JSX любой новой секции (это уже в Skills checklist выше — здесь усиливаю причину: без него по умолчанию вылезает shadcn-дефолт, и я этого не вижу пока Володя не ткнёт).
5. **Память хранит решения проекта и феедбек, не Tailwind-классы и не API чужих библиотек.** Договорённости, доменные правила veo55, мои промахи и корректировки — норм. Сигнатуры функций, последние UI-паттерны — нет.

**Anti-pattern (как НЕ делать):** «накатаю карточку, классы Tailwind помню, пропсы Embla примерно такие, серый плейсхолдер «фото скоро»». Это и есть «иногда очень посредственный UI» которого хочу избегать. Открыть Playwright + WebFetch стоит 2-3 tool-вызова и 30 секунд — переделка после ревью стоит часы.

**Правила миграции:**

1. **Идемпотентность.** Скрипт можно прогнать дважды — итог одинаковый. Используется upsert по стабильному ключу (slug, externalId).
2. **Никакого ручного копи-паста контента из браузера в Payload-админку.** Это техдолг — потерял источник правды, не воспроизведёшь.
3. **Сначала dry-run на dev БД, потом prod.** Лог количества записей, sample-выборка глазами.
4. **Скрипт фиксируется коммитом + краткая запись в `docs/migrations/<date>-<source>.md`** — что забрали, откуда, сколько записей, какой adapter.

### Обобщение блоков под `_template` — с самого начала, не «потом извлечём»

Цель проекта — `_template` (см. вверху файла). Поэтому **каждый блок при создании** проходит фильтр: «как он назовётся в шаблоне, чтобы автосервису/клинике/кофейне тоже подошёл?». Это R5++ нейминг (см. `.claude/rules/common.md`).

| ❌ Не так | `VkFeedBlock`                                | `client/blocks/niche/`                                              | Доменное имя, в `_template` не уйдёт, в кофейню/клинику не переедет. ### Технический долг — честно и сразу, не «потом»

- **Если ввели техдолг — пишем `// TODO(holygrail-<номер>):` или явный коммент в коде и фиксируем в `plan/PLAN` как отдельную задачу.** Не «потом вспомним».
- **Если знаем что надо рефакторить кусок чтобы фича села правильно — рефакторим сейчас**, не «потом, оно работает».
- **Никаких хаков с пометкой «временное» которые остаются на год.** Временное → либо сразу убираем, либо превращаем в нормальное решение, либо явный `TODO` с тикетом.

### Тесты — сразу, где нужны, не «потом покрою»

- **Контракты, рендер блоков, валидация форм, нишевая логика (фильтры/поиск Dogs/Litters)** — юнит-тест **в том же коммите**, не «следующим PR».
- Не покрытие ради метрики. Покрываем то что **реально может сломаться** и где **поведение должно быть зафиксировано** (контракт = тест).

### Документация в коде — обязательно для публичного API

> Это override дефолтного поведения Claude Code, который по умолчанию комментариев не пишет. В Web Holy Grail — пишем.

- **Модели данных (типы `contracts/`, Payload-коллекции, доменные сущности)** — JSDoc / TSDoc с описанием поля: что это, в каких единицах, чему может быть равно (`'draft' | 'published'`), что обязательно, что опционально.
- **Публичные методы / экспортируемые функции / React-компоненты** — TSDoc-блок с описанием: что делает, что принимает, что возвращает, какие side effects. Не «WHAT» (это и так видно), а **назначение + контракт поведения**.
- **Сложные алгоритмы и нетривиальные места** — коммент **«ПОЧЕМУ так»**, не «что делает». Что делает — читается из кода, почему — пропадает с автором.
- **Нишевая логика (`Dogs`, `Litters`, родословные, поиск)** — кратко описать доменные правила в коде: «помёт = группа щенков с одной матерью, max 1 active litter at a time», и т.п.
- **Хаки и workaround'ы** — обязательно коммент с причиной, ссылкой на issue/обсуждение и условиями когда можно убрать.

**Внутренние частные функции** — комментировать только когда WHY не очевидно из имени и тела. Не «// returns the user» к `getUser()`.

## Рабочий цикл сессии (operational discipline) — после load-context

> Это **не разовое**, это операционный ритм каждой сессии. Володя сформулировал явно: «после постановки задачи и загрузки контекста — так и работать». Все семь пунктов работают вместе; пропустишь один — теряется доверие и контроль.

1. **Сохранять и вести контекст в памяти.** Источник правды — MCP `HolyGrail/session/active` (что в работе ПРЯМО СЕЙЧАС) и `HolyGrail/plan/PLAN` (этапы и статусы). Не держать состояние только в голове / только в текущем чате. При крахе сессии следующая Claude восстанавливает картину чтением этих заметок — за минуту, без переспроса.

2. **План пишется пунктами линейно** — этапы (`A` / `B` / `C`) → подэтапы (`A1`, `A2`, …) → атомарные действия. Источник правды — `HolyGrail/plan/PLAN`, не план-в-уме, не план-в-комментарии. Каждый пункт читабельный без контекста («A1. Зачистить Dogs `chipsa`/`bars` из dev БД» — самодостаточно). Без вложенностей глубже 3 уровней.

3. **Расширение плана — только при новом запросе.** Новое требование Володи → **добавить пункт** в существующий план там, где он логично ложится по приоритету. Не «переписать всё». Старое не удалять — переносить в «отложено» с пометкой. Никогда не размывать активный фокус добавлением «попутных» задач, всплывших по дороге («заодно почищу X» — нет, в backlog).

4. **Приоритизация явная.** Один **фокус №1** в верху файла, всё остальное — упорядочено, помечено, но НЕ конкурирует за внимание сейчас. Любая новая задача — либо встраивается в фокус, либо ниже, никогда «вровень с фокусом».

5. **Не скакать с задачи на задачу.** Линейно: A1 → smoke (зелёный) → A2 → smoke → A3 → … Володя прерывает корректировкой посреди этапа — **актуализировать план** (внести правку в подэтап) + **продолжить текущее**, не пивотировать на половине фразы. См. `feedback_linear_work.md`, `feedback_no_guessing_rejections.md`. Если корректировка требует поменять активный пункт — менять явно, отметив в PLAN «доделано до X, переключаюсь на Y по запросу [time]».

6. **Прогресс обновляется автоматически через cron auto-tick — кронжоб создаю Я в начале каждой сессии после load-context.** Cron не «уже работает», он session-scoped и умирает с предыдущей Claude. Шаг в `/load-context` workflow: **`CronList` → если auto-tick не найден → `CronCreate`** на off-minutes (например `7,22,37,52 * * * *`, не `:00`/`:30`, чтобы не нагружать API на пиковых минутах). Prompt в `CronCreate`: «Тихо: прочитай `HolyGrail/session/active`, сделай `git log --oneline -5`, `TaskList`, добавь в секцию `## Progress (auto-tick)` строку `[YYYY-MM-DD HH:MM] commits=N tasks=done/total focus=<кратко>`. Если 30 минут без новых коммитов И без смены статусов задач — добавь маркер `⚠ stuck` и встрянь Володе одной фразой. Иначе молча обнови.» Это даёт Володе видимость прогресса без ручных отчётов от меня. `durable=false` — нормально для session-scoped (создам в следующей сессии заново); `durable=true` ставлю только когда явно нужно пережить рестарт.

7. **Сверка цель vs фактическая реализация.** На каждом завершённом этапе — короткий self-review: «что заявлено в PLAN на этапе X ↔ что реально в коде / БД / UI». Несовпадения — либо доделать, либо обновить PLAN с честной отметкой «откатил / не сделал / отложил» + причина. **Не отмечать как done если факт не совпадает** — Володя ловит это мгновенно (см. `feedback_analyze_before_doing` про «декларация ≠ действие»).

Этот цикл прогоняется **не в начале сессии**, а сразу **после** `/load-context` + первой постановки задачи. Дальше — рабочий ритм до конца сессии.

## Workflow

1. **Перед задачей:** прочитать релевантные заметки из Memory MCP (`HolyGrail`), сверить `plan/PLAN`, понять в каком этапе находимся.
2. **План в MCP, не в файле.** Этапы и статусы (todo/doing/done) живут в `plan/PLAN` (и подпланы в `plan/<этап>`). Обновлять по ходу через `edit_note`.
3. **План коммитов — заранее, до работы.** В подплане этапа выписать какие будут коммиты (1–3 на этап, до 5 на крупный, до 7 на очень сложный). Идти линейно: коммит 1 → зелёный → коммит 2 → зелёный. Не катиться вперёд и потом пытаться нарезать ретроспективно.
4. **Линейно.** Один этап за раз. Не начинать новое пока текущее не зелёное.
5. **Смоук после каждого куска.** Поднялось? Открылось? Не падает?
6. **Тесты — где есть логика.** Контракты, рендер блоков, формы. Не ради покрытия — ради того что реально может сломаться.
7. **Не строить выше того что нужно сейчас (R9).** Либа, зонтик, агентский слой высокого уровня — после нескольких сайтов, не до первого.

## Git-style (короткое — детали в скилле `/git-style`)

- **Атомарные коммиты, 1–3 на обычную задачу, до 5–7 на крупную.** Больше — признак что задачу пора резать на несколько PR.
- **Каждый коммит проходит тест `git checkout <sha>` → собирается и тесты зелёные.** Если коммит не проходит — склеить с соседним.
- **Не плодить `wip` / `fix` / `fix2` / `update after review` / `checkpoint`.** Хочется visibility прогресса — обновляй `plan/PLAN` в MCP, это правильное место для статуса, не git history.
- **Внутри одного логического слоя — `git commit --amend` (или `--amend --no-edit`),** не новый коммит.
- **Формат сообщения:**

  ```
  type(scope): краткий итог в форме результата

  - конкретный пункт 1
  - конкретный пункт 2
  - ... (5–12 пунктов)
  ```

  `type` ∈ {feat, fix, refactor, test, docs, chore}; `scope` = `client` / `cms` / `contracts` / `deploy` / `docs` / `claude` / `<блок>`.

- **Заголовок = итог задачи (результат), не описание процесса.** «add hero block» а не «implement hero block from design».
- **Без `Co-Authored-By`** (это глобальная преференция Володи).
- **История линейная, без merge-коммитов.** При обновлении ветки — rebase.

## Stop-conditions (зови Володю)

1. Архитектурная развилка (новая модель данных, breaking contract, выбор между двумя стеками).
2. Зацикливаюсь — один и тот же fix не работает второй раз подряд.
3. Деструктивная операция на чём-то shared (прод veo55, удаление веток, force-push).
4. Что-то выглядит подозрительно.

## Секреты — Infisical с первого коммита

Заложено до того как стало больно (см. PLAN, Шаг 3, коммит 2). На VPS деплой — один `infisical run -- docker compose up -d`.

- **`.env` НИКОГДА в git.** `.gitignore` строгий. Если случайно — пушу с force-удалением и ротирую секрет.
- **`.env.example`** — только шаблон-документация, в git.
- **Локально:** `infisical login` один раз, дальше `pnpm dev` под `infisical run --`.
- **В Docker:** обёртка через `infisical run` или передача через env-from в compose с подцеплением Infisical CLI на хосте VPS.
- **Тест:** `pnpm dev` без секретов должен фейлить с понятным сообщением, не работать на дефолтах.

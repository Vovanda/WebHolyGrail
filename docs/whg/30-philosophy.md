# Holy Grail — философия и архитектура (входной документ)

> Entry document. What the framework is and which rules it enforces, before any code is read.

---

## 1. Что это

WHG — opinionated dev-среда и растущая база компонентов. Не конструктор и не визуальный билдер; компоновка страниц происходит в коде (блоки), правка контента — через Payload-админку.

Свойства, которые держим:

- предсказуемые границы, типовые паттерны — junior contributors and LLM agents входит без разбега;
- потолка масштабирования нет — расширение **сбоку** (новый api, новый модуль, новая БД), не переписыванием ядра.

Одной строкой: **скучный предсказуемый каркас сейчас — безграничное расширение потом.**

## 2. Два горизонта (не путать)

**Сейчас (сроки):** берём готовое, свой CSS/примитивы не клепаем, цель — рабочий сайт в дедлайн.

**Потом (спецзаказы):** под нестандартное пишется свой контрол, оседает в базе компонентов по тому же паттерну. База растёт **органически из работы**, не отдельным проектом.

## 3. Стек

- **React + Next.js** — фронт (client): SSR/роутинг/SEO из коробки.
- **shadcn/ui** — база компонентов; копируется в проект, не чёрный ящик.
- **Tailwind** — стилизация, утилиты внутри компонентов (R1), без догмы.
- **CSS-переменные (токены)** — тема с первого дня.
- **Payload CMS** — контентный слой (cms): MIT, TypeScript, блочная модель, своя админка.
- **БД** — за слоем доступа, выбор прагматичен (SQLite визитке, Postgres растущему).
- **contracts** — разъём между фронтом и бэком; форма данных, источник правды.
- **api** (при росте) — свой бэк бизнес-логики (.NET/Node), через репозитории.

## 4. Правила (соблюдает каждый)

**R1. Не верстать страницы голыми утилитами.** Tailwind прячется внутрь компонентов; наружу — чистый типизированный интерфейс.

**R2. Токены конечны.** Цвета/отступы/радиусы — только из CSS-переменных. `bg-[#3a7bd5]` запрещён.

**R3. Изоляция фронта и бэка через контракт.** Фронт не знает бэк, граница — contracts.

**R4. Масштабирование сбоку.** Новая функция = отдельный сервис/БД/модуль + фронт-секция к его API через contracts. **Ядро не трогается.**

**R5. Секция/контрол/блок = чистая функция от пропсов, на любой глубине.** Главный риск долгой жизни — «умные» секции, лезущие в данные и копящие настройки → через год тот же неуправляемый комок, что SP Page Builder.

**R6. Без догм.** Inline `style` — для рантайм-динамики. Utility — для скорости. Раздельный CSS — для сложных контролов. Инструмент по бусту, не по идеологии.

**R7. Целевая нагрузка на старте — нулевая, у всех.** Никакого highload авансом. Highload-готовность — следствие чистой архитектуры, добавляется по факту, не заранее.

**R8. Доступ к данным инкапсулирован; конкретный слой — неважен.** К БД никто не лезет напрямую в обход слоя доступа. `api/` → репозитории. `cms/` (Payload) → его собственный слой доступа.

**R9. Абстракция следует за опытом, не предшествует ему.** Не строить обобщение раньше, чем появился конкретный опыт, который оно обобщает. Сначала одна работающая вещь — потом обобщение того, что повторилось. **Конкретное → обобщение → автоматизация**, не наоборот.

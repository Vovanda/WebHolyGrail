# Architectural rules (R1-R9)

> Правила соблюдаемые каждым разработчиком. Категория продукта, боль клиента/разработчика, что в коробке и roadmap — в [`00-overview.md`](00-overview.md). Stack rationale — в [`15-vision.md`](15-vision.md).

---

**R1. Не верстать страницы голыми утилитами.** Tailwind прячется внутрь компонентов; наружу — чистый типизированный интерфейс.

**R2. Токены конечны.** Цвета / отступы / радиусы — только из CSS-переменных. `bg-[#3a7bd5]` запрещён.

**R3. Изоляция фронта и бэка через контракт.** Фронт не знает бэк, граница — `contracts/`.

**R4. Масштабирование сбоку.** Новая функция = отдельный сервис / БД / модуль + фронт-секция к его API через `contracts/`. Ядро не трогается.

**R5. Секция / контрол / блок = чистая функция от пропсов, на любой глубине.** Блок не лезет в родителя, не тянет данные сам, не знает где стоит. Получает пропсы — рендерит детей.

**R6. Без догм.** Inline `style` — для рантайм-динамики. Utility — для скорости. Раздельный CSS — для сложных контролов. Инструмент по бусту, не по идеологии.

**R7. Целевая нагрузка на старте — нулевая.** Никакого highload авансом. Highload-готовность — следствие чистой архитектуры, добавляется по факту.

**R8. Доступ к данным инкапсулирован; конкретный слой — неважен.** К БД никто не лезет напрямую в обход слоя доступа. `api/` → репозитории. `cms/` (Payload) → его собственный слой доступа.

**R9. Абстракция следует за опытом, не предшествует ему.** Не строить обобщение раньше, чем появился конкретный опыт, который оно обобщает. Сначала одна работающая вещь — потом обобщение того, что повторилось.

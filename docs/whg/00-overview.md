# Overview

> Master doc. Что такое Web Holy Grail и зачем он нужен. Технические детали в соседних файлах: правила в [`30-philosophy.md`](30-philosophy.md), stack rationale в [`15-vision.md`](15-vision.md), структура репо в [`32-structure.md`](32-structure.md).

## Что это

**Архитектурный фундамент для self-hosted сайтов, которые растут.** Стек, структура проекта, границы между слоями и правила развития уже определены. Вы получаете не «ещё один шаблон с Next и Payload», а **систему ограничений, которая не даёт проекту сгнить по мере роста**.

Стартуете с визитки. Через полгода она становится визиткой с блогом. Через год — с каталогом услуг. Через два — с личным кабинетом клиента. Всё это **внутри того же репозитория, на том же фундаменте, без миграции на «настоящую платформу»**.

## Главная мысль

Сайты редко умирают из-за технологий. Они умирают из-за того, что были построены как временное решение. Дёшево и быстро — без расчёта на рост, без границ между фронтом и бэком, без места куда добавить новый функционал, не сломав старое.

WHG не пытается «сделать сайт» — для этого есть Tilda и тысячи Next.js-шаблонов. WHG **снимает с проекта риск тупика через год**: фундамент рассчитан так, чтобы новые задачи добавлялись сбоку, а не ломали ядро.

## Кому это

- **Разработчику-фрилансеру**, который уже переписывал сайт клиента с нуля и устал от одноразовых проектов.
- **Владельцу бизнеса**, который не хочет через год слышать «давайте всё переделаем, ваш сайт устарел».
- **Любому**, кому надоело каждый новый проект начинать с «какой выбрать стек / CMS / БД / деплой».

## Почему это работает

### Архитектурные решения уже приняты

| Слой            | Выбрано                     | Зачем зафиксировано                                           |
| --------------- | --------------------------- | ------------------------------------------------------------- |
| Frontend        | Next.js 15 + React 19       | Зрелый, массовый, любой разработчик подхватит                 |
| CMS             | Payload 3                   | TypeScript, schema-first, своя админка из конфига             |
| База            | SQLite → Postgres           | Один адаптер, нет миграции при росте                          |
| Хранилище медиа | S3-совместимое              | MinIO в dev, любой провайдер в prod                           |
| Секреты         | Infisical                   | Никаких `.env` на проде                                       |
| Deploy          | Docker compose + blue-green | Zero-downtime, откат за одну команду                          |
| Границы         | `contracts/` workspace      | Фронт и CMS общаются только через типы — независимая эволюция |

Вам больше не нужно это выбирать. Архитектурные споры закрыты до того, как вы открыли репо.

### Девять правил, защищающих проект от хаоса

Технологии — это половина проблемы. Вторая половина — **дисциплина роста**. Без неё через полгода в проекте появляется «умный» компонент, который знает про всё, через год — циклические зависимости, через два — желание всё переписать.

WHG прописывает R-правила (R1–R9 в [`30-philosophy.md`](30-philosophy.md)) — короткий свод архитектурной дисциплины. Несколько ключевых:

- **R3.** Фронт и бэк изолированы через контракты. Не «по соглашению», а структурой workspace.
- **R4.** Масштабирование сбоку. Новая фича = новый модуль, не правка ядра.
- **R7.** Никакого highload авансом. Решаем реальные проблемы, не воображаемые.
- **R9.** Абстракция следует за опытом. Сначала рабочее решение, потом обобщение.

Это не «code style». Это набор инженерных решений, которые предотвращают типичные болезни долгоживущих проектов.

## Сценарии роста

Сайт scaffold'ится с одной из стартовых точек — но архитектура одинаковая, отличается только стартовая конфигурация. Подробности — [`37-scaffolding.md`](37-scaffolding.md).

| Type                      | Что добавляется к базе                                                      |
| ------------------------- | --------------------------------------------------------------------------- |
| `minimal` (доступен)      | Pages / Media / Users / FormSubmissions / ReusableBlocks + пустая home page |
| `business-card` (roadmap) | + Pages presets, контактная форма, services grid                            |
| `blog` (roadmap)          | + Posts/Comments, `/blog` routes, sample post                               |
| `portal` (roadmap)        | + Customer Users (auth + roles), `/login`/`/dashboard`                      |

После запуска сайт продолжает расти через `blocks/domain/<niche>/` — независимо от того с какого type начали.

## Что в коробке (детали для уже принявших решение)

- **UI:** shadcn/ui + Tailwind + дизайн-токены под бренд.
- **CMS с админкой на русском:** drag-and-drop блоков, типизированные поля.
- **Изоляция БД:** код общается через слой доступа. SQLite на старте, Postgres одной строкой адаптера.
- **Хранилище секретов:** Infisical (self-host или cloud).
- **S3-совместимое хранилище медиа:** MinIO в dev, любой провайдер в prod.
- **Модульная архитектура:** фронт / CMS / контракты — отдельные workspace.
- **Blue-green deploy:** Docker compose + nginx + автоматический Let's Encrypt.
- **Sync template:** улучшения upstream подтягиваются скриптом, не трогая ваш доменный код.

## Layout как данные

Композиция страницы (где Header, где Sidebar, где Footer) хранится в CMS как JSON, не в JSX. Шесть фиксированных слотов — `top`, `bottom`, `left`, `right`, `center`, `overlay` — покрывают все типовые композиции. Поменять расположение элементов = поменять конфиг в админке. Без deploy.

Каждый блок объявляет видимость по брейкпоинтам (desktop / tablet / mobile) — контент-менеджер скрывает блок на мобиле чекбоксом.

## Roadmap

**Визуальный конструктор страниц поверх того же стека.** UX-слой над текущей CMS-моделью:

- **Разработчику** — собирать страницы из готовых блоков быстрее: canvas-режим вместо form-режима, live-preview, layout-builder для слот-композиции (сейчас в JSON, без UI).
- **Владельцу бизнеса / контент-менеджеру** — компоновать страницы из существующих блоков без redeploy.

Self-hosted, MIT. Конструктор будет частью того же шаблона.

**Что и в будущем остаётся за разработчиком:** стек, deploy, новые типы блоков (компоненты), доменная логика, интеграции.

## Что не покрывает

- **Visual builder с произвольной вложенностью блоков на canvas.** Payload даёт flat-список секций и составные блоки — этого хватает на типовые сайты. Полный builder уровня Tilda/Webflow — отдельный продукт. См. [`36-block-coverage.md`](36-block-coverage.md).
- **E-commerce-first.** Корзина / checkout / payments в core нет.
- **Hosting platform.** Bring-your-own-server. Blue-green compose included, но мы не Vercel/Netlify.
- **Multi-tenant.** Один инстанс = один сайт.
